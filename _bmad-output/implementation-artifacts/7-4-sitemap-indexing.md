## Story

As a **search engine crawler**,
I want an up-to-date XML sitemap and proper indexing controls,
So that all important pages are discovered and indexed efficiently.

## Acceptance Criteria

1. **Given** the web platform has product, content, and category pages
   **When** a crawler requests `/sitemap.xml`
   **Then** a dynamically generated XML sitemap lists all public pages (FR36)
   **And** the sitemap includes: product pages, content pages, category pages, static pages (home, about, content listing, products listing)
   **And** each entry has: `<loc>`, `<lastmod>`, `<changefreq>`, `<priority>`

2. **Given** products and content pages exist in the database
   **When** the sitemap is generated
   **Then** products have `<changefreq>daily</changefreq>` and `<priority>0.8</priority>`
   **And** content pages have `<changefreq>weekly</changefreq>` and `<priority>0.7</priority>`
   **And** `<lastmod>` reflects the actual `updated_at` timestamp from the database

3. **Given** the sitemap includes product and content URLs
   **When** products are added/removed (webhook sync) or content is published
   **Then** the sitemap updates when `bun run generate:sitemap` is re-run
   **And** only available products (`available = true`) and published content (`status = 'published'`) are included

4. **Given** the total number of URLs may grow over time
   **When** the URL count exceeds 50,000
   **Then** the script generates a sitemap index file (`sitemap.xml`) pointing to sub-sitemaps
   **And** each sub-sitemap contains at most 50,000 URLs
   **And** the sitemap index follows the sitemaps.org protocol

5. **Given** `robots.txt` controls crawler access
   **When** a crawler reads `/robots.txt`
   **Then** public pages are allowed for crawling
   **And** `/auth/`, `/checkout/`, `/cart/`, `/account/`, `/order/` routes are blocked
   **And** the `Sitemap:` directive points to the correct sitemap URL

6. **Given** the project uses a build script approach
   **When** the developer needs to regenerate the sitemap
   **Then** `bun run generate:sitemap` produces the updated sitemap
   **And** the script can run without Supabase credentials (gracefully degrades to static pages only)

7. **Given** Google Search Console integration
   **When** an administrator wants to submit the sitemap
   **Then** documentation exists in the script's JSDoc explaining manual submission steps

## Dev Notes

### Critical Architecture Constraints

- **Build script approach (NOT API route)** — TanStack Start v1 does NOT support `createServerFileRoute` for custom HTTP handlers (XML responses). The `tanstackStart()` Vite plugin only produces React-rendered HTML pages. The existing `scripts/generate-sitemap.ts` script generates a static `sitemap.xml` file at build time. **You must extend this script, not create an API route.** The script comment explicitly says: "Story 7.4 will replace this with a dynamic endpoint once the framework supports it" — but the framework STILL doesn't support it, so keep the script approach.

- **Supabase service role key required for build-time queries** — The script uses `SUPABASE_SERVICE_ROLE_KEY` to bypass RLS. This is correct for a build script (not client-facing). The content_pages table has RLS that filters `status = 'published' AND published_at <= now()`, but the script should add its own filters for defense in depth (matching the pattern in `fetchProducts()`).

- **Vanilla CSS + BEM only** — No CSS changes needed (this is a build script + robots.txt story).

- **No new dependencies** — Use only `@supabase/supabase-js` (already a dependency), `node:fs`, and `node:path`.

- **Graceful degradation** — If Supabase credentials are missing, the script must still produce a valid sitemap with static pages only (existing behavior — preserve it).

### Existing Code Patterns to Follow

```typescript
// Current fetch pattern in generate-sitemap.ts (follow same error handling style):
async function fetchContentPages(): Promise<ContentRow[]> {
  // Same Supabase client init pattern as fetchProducts()
  // Same env var fallback pattern
  // Same error handling (console.warn + return empty array)
  const { data, error } = await supabase
    .from("content_pages")
    .select("slug, updated_at")
    .eq("status", "published")
    .lte("published_at", new Date().toISOString())
    .order("updated_at", { ascending: false });

  if (error) {
    console.error("❌ Content pages query failed:", error.message);
    return [];
  }
  return data ?? [];
}
```
