## Session Overview

**Topic:** Plateforme d'affiliation white-label — site web + apps mobiles (Expo/React Native) intégrant les catalogues produits de marchands partenaires avec un parcours d'achat complet sans redirection, rémunération par commissions d'affiliation.

**Goals:**

- Comprendre les options d'API d'affiliation white-label existantes
- Explorer comment réaliser un checkout intégré sans redirection
- Identifier les défis juridiques/fiscaux (statut non-commerçant)
- Définir l'architecture technique (catalogues, panier, paiement, suivi S2S)
- Générer des idées créatives pour se différencier sur le marché

**Concept Breakthrough:**

> **"Personal Shopper Digital"** — une plateforme d'affiliation qui offre une expérience e-commerce premium avec une structure de coûts de blog affilié. L'utilisateur vit une expérience type boutique en ligne haut de gamme. En coulisses, zéro stock, zéro logistique, zéro statut commerçant. La différenciation vient de l'UX et de l'IA, pas du produit.

**Motivation fondatrice:**

> "Créer mon propre site de e-commerce, facile à gérer et à maintenir, avec le moins de contraintes possibles, tout en me démarquant des autres."

### Session Setup

Le projet vise à créer une plateforme qui agrège les produits de marchands partenaires via des API d'affiliation, en offrant une expérience utilisateur totalement intégrée (white-label). L'utilisateur ne quitte jamais la plateforme pour naviguer, ajouter au panier, ou finaliser son achat. Le suivi des commissions se fait en server-to-server (S2S). Le porteur de projet ne souhaite pas avoir le statut de commerçant.

## Phase 2 — Six Thinking Hats (Reconnaissance de Patterns)

### Analyse Multi-Perspective

**Blanc (Faits):** Ratio élevé de suppositions vs faits connus. Le business analyst doit convertir les hypothèses en données concrètes, notamment sur les fournisseurs white-label.

**Rouge (Émotions):** Motivation profonde = "Créer mon propre site de e-commerce, facile à gérer et à maintenir, avec le moins de contraintes possibles, tout en me démarquant des autres." ADN = Autonomie + Simplicité + Différenciation.

**Jaune (Avantages):** Top 5 — (1) Modèle prestation de service = légèreté maximale, (2) Search IA conversationnelle = différenciation immédiate, (3) TanStack + Expo = vitesse et contrôle, (4) White-label invisible = expérience premium, (5) Philippines + marché mondial = arbitrage coûts/revenus.

**Noir (Risques):** Un seul risque existentiel = trouver les fournisseurs. Risques secondaires gérables : statut juridique ambigu, ambition MVP élevée, confiance jour 1, dépendance fournisseurs.

**Vert (Solutions):** Plan B sur chaque risque. Architecture découplée des fournisseurs. Adapter Pattern pour la flexibilité.

**Bleu (Vue d'ensemble):** 4 piliers (UX, Tech, Business, Fournisseurs). Insight stratégique : recherche fournisseur = prérequis, pas tâche parallèle.

### Patterns Identifiés

- Le projet est viable SI les fournisseurs existent — c'est binaire
- L'architecture doit être fournisseur-agnostique (modulaire, pluggable)
- Le "wow effect" vient de l'UX + search IA, pas des features sociales
- La confiance se construit par le design et le contenu, pas par les badges

## Phase 3 — First Principles Thinking (Développement des Idées)

### Vérités Fondamentales

**#1 — L'e-commerce réduit à l'essentiel :** Trouver → Choisir → Payer → Recevoir. Deux étapes sous contrôle total (afficher + choisir), deux dépendent du fournisseur (payer + recevoir). Paiement = scénario A (API commande) ou B (widget pluggé), JAMAIS C (encaisser soi-même).

**#2 — La vraie différenciation :** Le produit de Charles n'est pas le produit physique. C'est l'expérience d'achat — la navigation, la search IA, l'agrégation, la curation, la confiance UX. Modèle "personal shopper digital".

**#3 — Architecture minimale :** 3 zones — Frontend (TanStack + Expo), Backend (Supabase + Search IA), Fournisseurs (pluggables via Adapter Pattern). Interface unifiée entre backend et fournisseurs = indépendance totale.

**#4 — Séquence incompressible :**

1. Trouver un fournisseur (business analyst + pistes Charles) — **PRÉREQUIS**
2. Comprendre ses API/contraintes
3. Concevoir l'adapter + interface unifiée
4. Construire le MVP avec vrais produits
5. Lancer, mesurer, itérer
6. Ajouter d'autres fournisseurs

**Décision critique :** Zéro données mock. Développement uniquement avec de vraies API fournisseur. La recherche fournisseur est un prérequis, pas une tâche parallèle.

**Développable en parallèle (sans fournisseur) :** Design system, architecture technique (monorepo, configs), authentification, structure BDD, prototype moteur de recherche IA.
